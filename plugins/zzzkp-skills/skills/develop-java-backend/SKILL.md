# Java 后端开发

## 目标

在不破坏项目既有架构的前提下，实现职责清晰、事务明确、数据访问安全且便于维护的 Java 后端代码。

## 工作流

1. 读取 `pom.xml`、`build.gradle`、公共模块、配置类、异常处理、权限机制和 ORM 依赖。
2. 如果项目根目录存在 `BACKEND_DEVELOPMENT.md`，先读取其中的项目差异规则。
3. 阅读目标模块的 Controller、Service、Mapper/Repository、实体、DTO、VO、SQL/XML 和表结构。
4. 确认业务边界、请求与响应结构、数据来源、事务范围和权限要求。
5. 优先复用项目已有基础类、工具、枚举、对象转换和持久层能力。
6. 只修改本次任务相关代码，不顺手重构无关历史模块。

## 分层规则

- Controller 负责请求接收、参数入口和返回包装，不承载复杂业务编排。
- Service 负责业务校验、事务、流程编排、对象转换和跨组件协作。
- Mapper、Repository 和 DAO 只负责数据访问，不承载业务判断。
- 新功能不要继续堆入巨型 Controller 或 Service；在本次范围内建立清晰业务边界。
- 涉及数据库表时，使用明确实体、领域对象或参数对象承载结构，避免无边界传播原始 `Map`。
- 固定状态、类型、动作和来源优先复用或定义枚举，避免散落魔法字符串。

## 数据与事务

- 遵循项目现有 MyBatis、MyBatis-Plus 或 JPA 用法，不混入新的持久层风格。
- 简单单表操作优先复用 ORM 基础能力；复杂查询按项目惯例使用 XML、Wrapper、Specification 或查询对象。
- 使用参数绑定，禁止拼接用户输入生成 SQL。
- 避免 N+1 查询、不必要全量加载和无业务意义的排序。
- 写操作在 Service 边界设置清晰事务，并注意 Spring 同类自调用导致事务失效的问题。
- 避免在事务中执行远程调用、大文件 IO 或其他耗时操作。
- 批量写入、导入和导出按项目习惯控制批次、分页与内存占用。

## 复用与依赖

- 优先复用 Lombok、Hutool、Apache Commons、Guava 和项目公共工具，不重复手写样板或转换逻辑。
- 只有重复逻辑确实属于当前模块且没有现成实现时才新增工具类。
- 不主动引入新库；确需新增依赖时先说明原因、影响和替代方案。
- 文件上传和存储遵循项目现有文件系统、对象存储或数据库设计，不擅自改换存储方案。

## 项目规则与完成检查

- 项目专属规则写入项目自己的 `BACKEND_DEVELOPMENT.md`，不沉淀到本 Skill。
- 只有用户明确要求时才创建或修改项目规则文件。
- 检查参数校验、权限、数据权限、日志、脱敏、异常处理和返回格式。
- 检查职责边界、事务范围、SQL 安全、批量处理和无关修改。
