# Java Backend Development

## 目标

按目标项目既有习惯开发或重构 Java 后端接口，同时提升后续维护友好度：先确认项目依赖和公共能力，再基于已有框架开发；分层职责清晰，数据库表有明确实体承载，固定状态和类型优先枚举化，重复转换逻辑优先工具化，新增实现不破坏项目原有框架和代码风格。

## 使用范围

- 适用于 Java 后端接口新增、接口重构、数据库表映射、分层代码整理、枚举和工具类抽取。
- 适用于 Spring Boot、MyBatis、MyBatis-Plus、JPA 等常见 Java 后端项目，但必须以当前项目既有约定为准。
- 不适用于前端开发、纯接口文档编写、非 Java 后端项目，除非用户明确要求只参考其中的通用设计原则。

## 快速流程

1. 先识别当前项目、模块路径、技术栈、依赖清单和既有分层习惯，重点阅读 `pom.xml`、`build.gradle`、公共模块、配置类、已有工具类和 ORM 相关依赖。
2. 若目标项目根目录存在 `BACKEND_DEVELOPMENT.md`，先读取该文件；项目级说明只补充项目差异、覆盖项和落地约定，未提及事项继续遵循本文件通用规则。
3. 阅读目标模块已有的 Controller、Service、Repository/Mapper、实体、配置、SQL/XML 或表结构脚本。
4. 若涉及数据库表，先确认或补齐实体/领域对象，再改 Service 或持久层。
5. 若存在状态、类型、动作、来源等固定值，优先新增或复用枚举。
6. 若出现重复参数转换、判空、列表解析、数值转换、格式化等逻辑，先查项目已有依赖和公共工具类，例如 Hutool、Apache Commons、Guava、项目 `common-utils` / `common-core`，确无合适实现时再抽模块内工具类。
7. 完成后用静态搜索核对是否存在重复私有工具方法、散落字符串状态、无用持久层方法或违反项目约定的实现。

## 核心规范

- Controller 保持薄层：负责请求接收、参数入口、基础返回包装，不承载业务编排和复杂数据转换。
- Service 负责业务校验、事务边界、流程编排、对象转换和跨表/跨组件协作。
- 持久层只负责数据访问，不承载业务判断；复杂 SQL、动态查询或 ORM 用法遵循当前项目既有规范。
- 编写 SQL 时以查询效率优先，非业务必要不添加排序；多次数据库查询能合理合并时优先合并，减少数据库往返次数；其他 IO 操作也应在保持职责清晰和可读性的前提下尽量合并。
- 若项目已有 Lombok，实体、DTO、VO、参数对象等简单数据承载类优先使用 Lombok 减少样板代码；不要手写重复 getter、setter、构造器、`toString`，除非项目风格或框架限制要求手写。
- 涉及数据库表时，优先使用明确的实体、领域对象或参数对象承载结构，避免无边界地传播原始 `Map`。
- 查询场景可以按项目习惯保留弹性参数，例如 `Map<String, Object>`、查询 DTO 或 Specification；写场景进入 Service 后应尽快转换为实体或明确参数对象。
- 若项目存在 MyBatis-Plus，单表新增、按主键删除、按主键更新、简单批量保存等基础写操作优先使用 MyBatis-Plus 能力，例如 `BaseMapper<Entity>`、`IService` 或项目已有封装，避免重复手写 XML/注解 SQL。
- 若项目存在 MyBatis 或 MyBatis-Plus，复杂查询、动态条件、联表、聚合、分页、统计、导出等应按项目既有习惯选择 XML、注解 SQL、Wrapper 或查询对象；不要为简单单表写操作重复造持久层方法。
- 固定状态、类型、动作、层级、来源等值优先枚举化，不要散落魔法字符串或重复常量。
- 重复工具逻辑优先复用已有依赖和项目公共工具类，例如 Hutool、Apache Commons、Guava、`StringUtils`、`CollectionUtils`、`DateUtils`、`common-utils`；确需新增时集中到模块内工具类，不要在多个 Service 中复制私有方法。
- 写操作应设置清晰事务边界，并使用项目既有异常处理和错误返回方式。
- 若设计文件上传导入场景，优先把文件留存到服务器上，数据库仅保留文件路径，禁止存大量文件数据到数据库
- 不主动引入新库；确需新增依赖时，先说明理由并确认与项目技术栈兼容。
- 修改时遵循 SOLID 原则，避免把新逻辑堆入已有大方法或破坏现有职责边界。

## 项目专属规则

项目级特有规则应跟随目标项目，不沉淀到 skill 内。

- 若目标项目根目录存在 `BACKEND_DEVELOPMENT.md`，先读取并遵循其中约定。
- 若目标项目没有 `BACKEND_DEVELOPMENT.md`，只使用本文件通用规则，并优先从当前代码中归纳项目约定。
- 新增项目专属规则时，应写入目标项目根目录的 `BACKEND_DEVELOPMENT.md`，不要写入 skill 的 `references/`。
- `BACKEND_DEVELOPMENT.md` 只记录项目差异、覆盖项、命名约定、固定落地模式和特殊限制，避免重复本文件已有原则。

## 禁止事项

- 未确认项目既有风格前，直接套用某个项目的专属规范。
- 为了实现单个接口引入不必要的新框架、新依赖或大范围重构。
- Controller、Service、持久层职责混杂。
- 同一类参数解析、字符串转换、集合解析逻辑在多个类中复制。
- 已有 Lombok、Hutool、Apache Commons、项目公共工具类或 MyBatis-Plus 能力可满足需求时，仍手写大量样板代码或重复工具方法。
- 对固定状态和类型使用散落字符串。
- 修改无关文件或重写用户未要求调整的模块。
