# Project Document Management

## 目标

统一管理多个项目的文档：所有项目文档都存放在环境变量指定的根目录下，每个项目拥有独立目录；查阅、生成、修改文档时先定位项目，再按“项目总览 -> 模块总览 -> 专题文档”的层级渐进读取或维护文档，避免一次性加载或生成过大的单文件文档。

## 必要前置

- 必须通过环境变量读取文档根目录。
- 推荐环境变量名为 `PROJECT_DOCS_ROOT`；如果当前团队已约定其他变量名，优先遵循用户或项目上下文中的明确约定。
- 如果环境变量不存在、为空或路径不可访问，先向用户说明缺失项并要求提供或配置根目录。
- 如果用户未明确说明要处理哪个项目，必须先询问项目名，不能猜测、不能默认选择第一个目录。

## 目录约定

文档根目录下按项目分目录：

```text
${PROJECT_DOCS_ROOT}/
├── 项目A/
│   ├── README.md
│   ├── frontend/
│   │   ├── README.md
│   │   └── 页面路由说明.md
│   ├── backend/
│   │   ├── README.md
│   │   └── xx外部接口使用说明.md
│   ├── database/
│   │   └── README.md
│   └── deployment/
│       └── README.md
└── 项目B/
    ├── README.md
    ├── api/
    │   └── README.md
    └── operations/
        └── README.md
```

- 项目目录名以用户指定项目为准。
- 项目根 `README.md` 是项目总览和一级索引，不承载模块细节。
- 一级模块使用目录承载，例如 `frontend/`、`backend/`、`database/`、`api/`、`deployment/`、`operations/`。
- 模块目录下的 `README.md` 是模块总览和模块内索引，例如 `backend/README.md` 介绍后端整体。
- 模块内长内容按专题继续拆分，例如 `backend/xx外部接口使用说明.md`。
- 不为同一主题创建多个近似文件或目录；新增前先检查项目根索引、模块索引和已有专题文档。

## 路径引用约定

- 项目根 `README.md` 必须注明项目代码路径，用于统一解释文档中的代码文件路径。
- 文档中引用项目代码文件、代码目录、模块路径时，必须以项目代码路径为基准写相对路径。
- 如果项目代码路径是 `D:/workspace/project-a`，代码文件应写为 `src/main/java/.../UserController.java`，不要写成相对文档根目录的路径。
- 文档之间的 Markdown 链接仍然以当前 Markdown 文件所在目录为基准，例如模块内链接写 `xx外部接口使用说明.md`，跨模块链接按相对位置写 `../database/README.md`。
- 不在专题文档中反复写绝对代码路径；如需说明基准，引用项目根 `README.md` 中的项目代码路径。

## 快速流程

1. 读取文档根目录环境变量，确认路径存在。
2. 确认用户指定的项目名；未指定时先询问。
3. 定位项目目录；不存在时根据任务判断：
   - 查阅或修改文档：先告知项目目录不存在，并询问是否需要创建或是否项目名有误。
   - 生成新项目文档：创建项目目录和项目总览文档。
4. 先读取项目目录下的 `README.md` 或文件列表，理解已有一级模块结构。
5. 根据用户需求读取对应模块的 `README.md`；只有需要细节时再读取模块内专题文档。
6. 生成或修改文档时，优先更新已有模块总览或专题文档；确需新增模块或专题文档时，同步更新对应层级的索引。
7. 完成后检查是否产生重复主题、巨型文档、断链索引或与项目目录不一致的文件。

## 渐进式披露规则

- 项目根 `README.md` 只包含项目简介、一级模块索引、关键入口、维护说明和必要摘要。
- 模块 `README.md` 只包含模块整体介绍、模块内专题索引、关键约定和必要摘要。
- 详细内容必须拆分到模块内专题文档。
- 一个专题文档只覆盖一个主要主题；主题扩大时继续拆分。
- 面向不同读者或场景的内容优先拆分，例如开发、部署、运维、数据库、接口、需求、排障。
- 查阅文档时先读项目根索引，再读模块索引，最后按需求读取最相关的一到数个专题文档。
- 修改文档时只改相关层级的文档和必要索引，不顺手重写无关模块或专题。

## 项目根 README.md 建议结构

```markdown
# 项目名称

## 项目信息

- 项目代码路径：D:/workspace/project-a

## 项目概览

简要说明项目目标、边界和当前状态。

## 文档索引

- [前端说明](frontend/README.md)
- [后端说明](backend/README.md)
- [数据库说明](database/README.md)
- [部署说明](deployment/README.md)

## 维护约定

说明文档拆分、命名和更新规则。
```

## 模块 README.md 建议结构

```markdown
# 后端说明

## 模块概览

简要说明后端模块职责、技术栈、边界和关键约定。

## 专题文档

- [xx外部接口使用说明](xx外部接口使用说明.md)
- [认证鉴权说明](认证鉴权说明.md)

## 维护约定

说明模块内专题文档的拆分和更新规则。
```

## 生成文档规范

- 先确认目标项目和文档主题。
- 先查看项目根 `README.md`、相关模块 `README.md` 和已有专题文档，避免重复创建。
- 新增一级模块时创建模块目录和模块 `README.md`，并更新项目根 `README.md`。
- 新增模块专题文档时放在对应模块目录下，并更新模块 `README.md` 的专题索引。
- 新增专题文档时，文件名使用清晰、稳定、可搜索的名称；优先遵循项目已有命名风格。
- 新增文档内容应包含明确标题、适用范围、核心内容和必要引用。
- 新增文档后必须更新对应层级的文档索引。
- 不把前端、后端、数据库、部署、接口等全部内容塞进一个 README。

## 修改文档规范

- 修改前先读取目标文档当前内容、项目根 `README.md` 和对应模块 `README.md` 索引。
- 保留项目已有术语、标题层级、链接格式和命名风格。
- 只更新与用户需求相关的段落；必要时调整索引。
- 如果发现内容应该拆分，先说明拆分理由，再迁移到对应模块目录下的专题文档。
- 删除或重命名文档时，必须同步修正所有相关链接。

## 查阅文档规范

- 先通过项目根 `README.md` 或文件列表判断相关模块。
- 再读取相关模块的 `README.md` 判断是否需要专题文档。
- 按用户问题只读取必要模块和必要专题文档。
- 回答时说明信息来自哪个项目和哪些文档。
- 如果索引缺失或无法判断相关文档，先列出可用模块和专题文档并选择最可能相关的少量文件阅读；仍不确定时向用户确认。

## 禁止事项

- 用户未指定项目时直接查阅、创建或修改某个项目目录。
- 环境变量缺失时自行在当前仓库或任意路径创建文档。
- 一次性读取整个文档根目录或所有项目文档。
- 把多个大主题合并进单个巨大 README。
- 把代码文件路径写成相对文档根目录的路径。
- 在专题文档中反复写绝对代码路径，除非用户明确要求。
- 为同一主题创建重复文档或目录，例如同时存在 `backend/`、`后端/`、`server/` 且内容重叠。
- 修改与用户需求无关的项目文档。
- 在 skill 内沉淀具体项目的业务资料；项目资料只写入对应项目目录。
