# Web 前端开发

## 目标

在不破坏项目既有架构和视觉风格的前提下，实现职责清晰、状态完整且具备基础响应式能力的 Web UI。

## 工作流

1. 读取 `package.json`、构建配置、页面目录、组件目录、样式方案、请求封装和同类页面。
2. 如果项目根目录存在 `FRONTEND_DEVELOPMENT.md`，先读取其中的项目差异规则。
3. 确认路由、状态管理、UI 组件库、CSS 方案和公共组件。
4. 划分页面结构、组件职责、请求逻辑、状态逻辑和样式范围。
5. 优先复用已有布局、表格、分页、表单、弹窗、loading、空状态和请求工具。
6. 只修改任务相关文件，并检查常见视口下的布局和交互状态。

## 实现规则

- 保持结构、样式、交互和数据处理职责清晰；复杂页面拆分为子组件、hooks、composables 或工具模块。
- 复用项目已有组件和设计 token，不为单一页面引入新的 UI 库、状态库或 CSS 框架。
- 使用 flex、grid、相对单位、`minmax()`、`clamp()` 和媒体查询实现基础响应式布局。
- 固定像素只用于边框、图标和设计系统已有的小尺寸值，不锁死页面主体宽高。
- 请求状态至少区分加载、成功、空数据和失败；提交操作处理禁用、防重复点击和错误反馈。
- loading 优先局部化，只有整页首次初始化时才考虑全局 loading。
- 当数据规模未知或可能增长时，为表格和列表设计分页、懒加载或虚拟列表方案。
- 筛选、搜索或排序条件改变时，按业务规则重置分页状态。
- 复用现有格式化、校验、权限和数据转换工具，避免跨页面复制逻辑。
- 遵循 SOLID 原则，不顺手重构用户未要求的模块。

## 项目规则

- 项目专属组件库、断点、目录和命名约定写入项目自己的 `FRONTEND_DEVELOPMENT.md`，不写入本 Skill。
- 只有用户明确要求沉淀项目规则时才创建或修改该文件。
- 项目说明未覆盖的内容继续遵循本 Skill，并以现有代码风格为最终依据。

## 完成检查

- 检查窄屏溢出、遮挡、不可操作控件和表格横向滚动。
- 检查 loading、空数据、错误、成功、禁用和重试状态。
- 检查是否重复实现已有组件或工具。
- 检查修改范围是否与用户请求一致。
