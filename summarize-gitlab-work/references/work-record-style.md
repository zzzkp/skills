# 工作记录归纳与表达规范

## 输出格式

按业务成果编号，不按提交时间逐条罗列：

```markdown
1.项目名称（模块名称与工作类型）：
工作描述。

2.项目名称（模块名称与工作类型）：
工作描述。
```

## 语言风格

采用事实清单型表达。每条只写能够从提交或 diff 中确认的对象、动作和变化：

```text
新增/修改/调整……；支持/增加/移除……。
```

不要求追加价值总结。删除不能由代码或数据直接证明的“提升效率”“增强能力”“保障运行”等结尾。

优先使用以下动作词：

- `新增`：代码中出现新的页面、接口、字段、筛选、图层或交互能力。
- `优化`：已有能力的布局、逻辑、显示方式或操作路径发生具体变化，并说明改了什么。
- `重构`：代码或页面结构发生明显调整，并写明调整后的结构或展示方式。
- `完成`：提交证据覆盖一项能力的主要实现，但不要据此推断已经上线。

避免单独使用以下空泛表达：

- 提升分析能力、提升使用效率、提升操作便捷性。
- 增强完整性、增强规范性、增强可追溯性。
- 保障稳定运行、赋能业务、形成闭环。
- 相关优化、若干调整、进一步完善。

只有存在性能数据、测试结果、故障记录或明确业务反馈时，才描述可量化效果。

## 归并示例

### 合并前后端提交

提交 A 新增配置弹窗与保存按钮，提交 B 新增查询、保存接口及数据库版本维护，应合并为：

```text
新增指标门限配置入口和编辑弹窗，支持查看、修改门限值及备注；新增配置查询、保存和权限校验接口。保存配置时，先将当前配置置为失效，再写入新配置，并校验提交项是否完整。
```

### 合并公共组件与同步提交

公共 GIS 组件新增专题图层，随后业务项目提交“统一GIS代码同步”，只输出一项成果，并说明相同图层代码已同步至业务项目，不重复列两条。

### 拆分不同模块

4G 高负荷门限配置与 5G 高倒流门限配置属于不同业务模块。即使实现模式相似，也应分别成条，保留各自指标范围和业务术语。

## 事实边界

- diff 仅出现接口和页面代码时，直接列出新增的页面、接口和数据处理逻辑，不写“已上线”或“形成完整闭环”。
- 仅出现测试代码或配置时，不自动写“测试通过”。
- 提交标题与代码不一致时，以代码变更为准。
- 大规模同步提交需与源仓库提交比对，识别重复文件和新增差异。
- 对无法判断的项目归属或业务含义，向用户确认，不编写多套猜测性表述。

## 输出质量检查

1. 是否覆盖所有实质性提交。
2. 是否把同一能力的前后端与同步提交合并。
3. 是否保留用户能感知的字段、筛选、弹窗、下钻、图层和权限变化。
4. 是否避免代码文件名、类名和提交 SHA 干扰业务表达。
5. 是否使用统一的项目名称和模块术语。
6. 每句话是否能对应具体提交、文件变更或代码逻辑。
7. 是否删除没有数据依据的价值总结和效果描述。
8. 是否避免“提升、增强、保障、赋能、闭环、相关优化、若干调整”等空泛措辞。
