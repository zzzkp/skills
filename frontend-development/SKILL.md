# Frontend Development

## 目标

按目标项目既有习惯开发或重构前端页面与组件，同时提升后续维护友好度：先确认项目技术栈、目录结构和公共组件，再基于已有框架开发；结构、样式、逻辑职责清晰；页面具备基础响应式能力但不做过度适配；数据加载有明确状态反馈，优先局部 loading；表格类数据默认具备分页能力；新增实现不破坏项目原有框架和代码风格。

## 使用范围

- 适用于 Web 前端页面新增、页面重构、组件开发、表格列表、表单交互、数据请求、状态反馈和样式整理。
- 适用于原生 HTML/CSS/JavaScript、Vue、React、Angular、Svelte、Vite、Next.js、Nuxt 等常见前端项目，但必须以当前项目既有约定为准。
- 不适用于 Java 后端开发、纯接口文档编写、移动原生应用开发，除非用户明确要求只参考其中的通用前端设计原则。

## 快速流程

1. 先识别当前项目、模块路径、技术栈、构建工具、路由方式、状态管理、CSS 方案和已有 UI 组件，重点阅读 `package.json`、页面目录、组件目录、样式目录、请求封装和已有同类页面。
2. 若目标项目根目录存在 `FRONTEND_DEVELOPMENT.md`，先读取该文件；项目级说明只补充项目差异、覆盖项和落地约定，未提及事项继续遵循本文件通用规则。
3. 优先复用项目已有组件、布局、请求工具、loading 组件、分页组件、弹窗、表单控件、表格组件和样式变量。
4. 开发页面前先划分结构、样式、逻辑职责：原生项目中拆分 HTML、CSS、JS 文件；框架项目中按项目习惯拆分组件、样式、hooks/composables、请求方法和状态逻辑。
5. 布局默认具备基础响应式能力，优先使用弹性布局、网格布局、相对单位和合理断点；不要用全局固定 `px` 堆出页面。
6. 涉及数据请求时，必须设计 `loading`、空数据、错误和成功状态；列表、表格、详情卡片等区域优先使用局部 loading。
7. 涉及表格或大列表时，默认考虑分页、分页状态、筛选条件、当前页重置、加载状态和空数据展示。
8. 完成后用静态搜索核对是否存在样式和逻辑混写、重复请求逻辑、缺失 loading、无分页表格、无意义固定宽高或违反项目约定的实现。

## 核心规范

- 页面结构、样式和交互逻辑保持职责分离。原生页面不要把大量 CSS 和 JavaScript 内联到 HTML；框架组件也不要把复杂请求、转换、校验和样式细节全部堆在单个大组件中。
- HTML 负责语义结构，CSS 负责视觉与布局，JavaScript 负责状态、交互和数据流；职责交叉时优先拆分为组件、工具函数、hooks、composables 或独立样式文件。
- 避免单一页面或组件过大。一个组件只承担清晰职责；当结构、状态、请求、表格、表单、弹窗或操作区开始膨胀时，优先拆分为子组件、组合函数、工具方法或独立模块。
- 具备通用性的 UI、交互、状态处理和数据展示能力应尽早封装为公共组件或公共逻辑，例如筛选栏、分页表格、状态标签、确认弹窗、空状态、局部 loading 容器和格式化方法。
- 响应式布局以项目实际需要为准，保证常见桌面和移动宽度下可用，不为了少量边界设备做过度复杂适配。
- 不全局依赖固定 `px` 实现布局。优先使用 `%`、`rem`、`em`、`fr`、`auto`、`minmax()`、`clamp()`、`max-width`、`min-width`、`flex`、`grid` 和媒体查询。
- 固定 `px` 可以用于边框、细小图标、阴影、极小间距或设计系统已有 token，但不要用大量固定宽高锁死页面主体布局。
- 文本、按钮、表单、表格和卡片在窄屏下不能互相遮挡；必要时允许换行、折行、横向滚动表格或切换为更适合小屏的布局。
- 数据请求必须有状态建模，至少区分加载中、成功、空数据和失败；不要让用户面对无反馈的空白区域。
- loading 优先局部化。表格刷新只遮罩或替换表格区域，卡片刷新只影响卡片区域，按钮提交只影响按钮或表单区域；只有整页首屏初始化才考虑全局 loading。
- 交互状态要完整，包括禁用、提交中、防重复点击、错误提示、空状态、重试入口和必要的成功反馈。
- 请求、数据转换、格式化、校验和权限判断优先复用项目已有工具和封装；确需新增时集中到明确模块，避免在多个页面复制私有方法。
- 修改时遵循 SOLID 原则，保持组件单一职责，避免把展示、请求、业务规则、数据转换和样式控制混在一个难以维护的文件中。
- 不主动引入新库；确需新增依赖时，先说明理由并确认与项目技术栈兼容。

## 表格与列表规范

- 表格类数据默认需要分页，除非数据量确定很小或项目已有明确例外。
- 分页状态应至少包含当前页、每页条数、总数和加载状态；按项目习惯支持页码切换、每页条数切换和刷新。
- 筛选、搜索或排序条件变化时，应按业务习惯重置到第一页，避免请求不存在或不符合预期的页码。
- 表格 loading 应为局部 loading，不应因为表格刷新导致整个页面不可用。
- 表格必须处理空数据和请求失败状态；空数据不要只显示空白。
- 大列表避免一次性渲染不可控数据量；项目已有分页、虚拟列表或懒加载能力时优先复用。
- 表格列宽和操作列应具备基本响应式策略；窄屏下可使用横向滚动、列裁剪、折叠信息或项目已有移动端表格方案。

## 项目专属规则

项目级特有规则应跟随目标项目，不沉淀到 skill 内。

- 若目标项目根目录存在 `FRONTEND_DEVELOPMENT.md`，先读取并遵循其中约定。
- 若目标项目没有 `FRONTEND_DEVELOPMENT.md`，只使用本文件通用规则，并优先从当前代码中归纳项目约定。
- 新增项目专属规则时，应写入目标项目根目录的 `FRONTEND_DEVELOPMENT.md`，不要写入 skill 的 `references/`。
- `FRONTEND_DEVELOPMENT.md` 可记录组件库、样式方案、响应式断点、分页组件、loading 组件、请求封装、目录结构、命名约定、固定落地模式和特殊限制，避免重复本文件已有原则。

## 禁止事项

- 未确认项目既有技术栈、目录结构、组件库和样式方案前，直接套用全新架构或个人习惯。
- 为了单个页面引入不必要的新 UI 库、状态管理库、CSS 框架或大范围重构。
- 把 HTML、CSS、JavaScript 大量写在同一个文件中，或在框架项目中把复杂页面堆成不可维护的单个巨型组件。
- 单个页面或组件持续堆积功能，不拆分明显独立的表格、表单、筛选、弹窗、操作区或通用状态逻辑。
- 多个页面重复实现相似组件、交互、loading、空状态、分页表格或格式化逻辑，而不抽取公共组件或公共方法。
- 全局使用固定 `px` 宽高实现主体布局，导致页面在不同视口下溢出、遮挡或不可用。
- 数据请求没有 loading、空数据、错误状态或重试策略。
- 表格和大列表无分页、无加载状态或一次性渲染不可控数据量。
- 同类请求、格式化、状态处理和表格分页逻辑在多个页面中复制。
- 修改无关文件或重写用户未要求调整的模块。
